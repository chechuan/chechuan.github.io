ABAC权限模型

發表於2020-11-19|更新於2023-10-19|智能高并发审批流系统
|閱讀量:

前面我们使用了RBAC权限模型,今天来介绍一下ABAC。
不要被它的名字吓到,每次我们听到”权限”这个词，先思考一下，什么是权限?为什么会有权限?权限是为了实现什么目的?

权限管理，一般指根据系统设置的安全规则或者安全策略，用户可以访问而且只能访问自己被授权的资源，不多不少。权限管理几乎出现在任何系统里面，只要有用户和密码的系统。 很多人常将“用户身份认证”、“密码加密”、“系统管理”等概念与权限管理概念混淆。

举几个例子

企业IT管理员一般都能为系统定义角色，给用户分配角色。这就是最常见的基于角色访问控制。场景举例：

1、给张三赋予“人力资源经理”角色，“人力资源经理”具有“查询员工”、“添加员工”、“修改员工”和“删除员工”权限。此时张三能够进入系统，则可以进行这些操作；

2、去掉李四的“人力资源经理”角色，此时李四就不能够进入系统进行这些操作了。

以上举例，局限于功能访问权限。还有一些更加丰富、更加细腻的权限管理。比如：

1、因为张三是北京分公司的“人力资源经理”，所以他能够也只能够管理北京分公司员工和北京分公司下属的子公司（海淀子公司、朝阳子公司、西城子公司、东城子公司等）的员工；

2、因为王五是海淀子公司的“人力资源经理”，所以他能够也只能够管理海淀子公司的员工；

3、普通审查员审查财务数据的权限是：在零售行业审核最高限额是￥50万，在钢铁行业最高限额是￥1000万；高级审查员不受该限额限制；

4、ATM取款每次取款额不能超过￥5000元，每天取款总额不能超过￥20000元。

这些权限管理和数据（可以统称为资源）直接相关，又称为数据级权限管理、细粒度权限管理或者内容权限管理。

三种模型:

ABAC权限模型是基于属性的访问控制

用户在携带自身的属性值包括时间属性，地域属性，人物属性，然后向资源发送请求，授权引擎 会根据subject所携带的属性进行判断，然后会给出拒绝或者同意的结果给用户，然后就可以访问资源。

详细步骤

ABAC授权的步骤

•1、用户访问资源，发送原始请求。

•2、请求发送到策略实施点（PEP），PEP构建xacml格式请求。

•3、PEP将xacml请求发送到策略决策点（PDP）。

•4、PDP根据xacml请求，查找策略管理点（PAP）中的策略文件。

•5、PDP从策略信息点（PIP）查找策略文件中需要的属性值（主体、资源、环境属性）。

•6、PDP将决策结果（permit、deny、不确定、不适用）返回给PEP。

•7、PEP发送请求到资源，并把资源返回给用户。

mysql错误: # django.db.utils.InternalError: (1054, “Unknown column ‘access.role_id_id’ in ‘where clause’”)解决办法

今天报了这样的错，第一反应看到后端status code 500，直奔后台看用户id,经过测试可以拿到,前端的用户id也跟后端的一样,表里的字段也一模一样。这就让我百思不得其解。问题出在哪?昨天折腾了一晚上，都没能找出来。今天跟同学聊天说起了这件事,他说你看你数据库里的字段是否正常。

第一种:

看你数据库的表的role_id字段,这里一般很难发现,由于这个字段是给用户加权限控制的时候加的字段,我的确把它忽略了。我们去Navicat里面将access(权限表)里的role_id改为role_id_id,点击保存。

django端models里的字段不变还是role_id。重新运行项目,执行删除操作成功,ye!!!

第二种:

相对比较简单粗暴,这也是我想到的第一种解决方案，既然逻辑删除无法删，那我们就用物理删除,导入pymysql,通过后端传过来的用户id进行删除，删除成功。
